Why Design Patterns?

  • Time and sequencing are often a core part of a game's architecture. Things must happen in the right order and at the right time.

  • Development cycles are highly compressed, and a number of programmers need to be able to rapidly build and iterate on a rich set of different behavior without stepping on each other's toes or leaving footprints all over the codebase.

  • After all of this behavior is defined, it starts interacting. Monsters bite the hero, potions are mixed together, and bombs blast enemies and friends alike. Those interactions must happen without the codebase turning into an intertwined hairball.

  • Performance is critical in games. Game developers are in a constant race to see who can squeeze the most out of their platform. Tricks for shaving off cycles can mean the difference between an A-rated game and millions of sales or dropped frames and angry reviews.

What is Good Software Architecture

  • Good architecture is about change. The measure of a design is how easily it accommodates changes. The system should be crafted in anticipation of change.

  • A major step to making changes in current codebases, is organizing and understanding the entire codebase already. Even then, the changes implemented have to be carefully understood and documented in anticipation of future changes.

  • Probably the most common flow chart in programming goes like this:
    Get Problem -> Learn Code -> Code Solution -> Clean Up... (repeat)

Introduce Decoupling

  • One issue with any software codebase is that they can grow beyond the scope of anybody being able to even learn the existing codebase. This makes it very hard to add to, change, or even organize it.

  • The first design pattern that could be applied here is the notion of Decoupling. The best way to think about decoupling is looking at two pieces of code that are coupled. Being coupled means that one can't be understood without the other and vise versa.

  • So this concept of Decoupling is seperating two ideas into their own independent and smaller ideas. Now you can think smaller and focused, also if one is relevant to your current problem, it is much easier to understand and apply.

  • Decoupling helps us accomplish an important task in good software -
    minimize the amount of knowledge you need to have, before being able to make progress.
    So the less coupling we have, the smaller the ripple effect a change will have.

Cost of Decoupling

  • Like gardening, it's not enough to put in new plants. You must also weed and prune. You have to decide what should be decoupled and put in the time to make it work.

  • Adding this layer of abstraction takes time to develop, debug, and maintain. And really, we're just speculating that we'll need that flexibility later on. This speculation can help or hurt in the long run depending on the guess.

  • Be careful not to turn the tables and fill up the entire architecture with layers of abstraction.

Performance & Speed

  • Can intricate architecture and abstraction take away from performance? Many patterns make code more flexible by relying on virtual dispatches, interfaces, pointers, messages, and other mechanisms that have at least some runtime cost.

  • We use these patterns to make our program more flexible but performance can take a hit considering how much abstraction we need. But performance is all about assumptions, and the process of optimization relies on concrete limitations. For best optimizations we need to know the exactly limitations of our program (num enemies, class types, etc.)

  • However, when considering development speed, flexibility is our guiding light. The faster we can try out ideas and experiment with the program, the more flexible our prototypes can become and reduce total development time. Likewise, optimization often makes code less flexible.

  • The takaway here is, it's easier to make a fun game fast than it is to make a fast game fun. So keep the code flexible until design settles down and tear out some abstraction to improve performance later on.

Program Goals

  • We have a few forces in play:
      1. We want nice architecture so the code is easier to understand over the lifetime of the project.
      2. We want fast runtime performance
      3. We want to get today's features done quickly.

  • These goals are all mainly focused on some kind of speed either regarding development or the program itself. While they are all partially in opposition, thinking about all three over the course of development can help overall.

  • There is no easy trade-off here, just different flavors of wrong. If there was a correct answer, everyone would do it. This is great though! We have the opportunity to find what works best for our personal style and can start mastering it.

Simplicity

  • One method of easing our program goal constraints is Simplicity. We want code that once read, is fully understand and provides a clear solution

  • This is a nuanced and difficult taks unfortunatly. Finding the simplicity in a program is like pattern matching or puzzle solving. It takes effort to see through the scattering of example use cases to find the hidden order underlying them all.

  • Blaise Pascal - I would have written a shorter letter, but I did not have the time.

Summary

  • Abstraction and decoupling make evolving your program faster and easier, but don't waste time doing them unless you're confident the code in question needs that flexibility.

  • Think about and design for performance throughout your development cycle, but put off the low-level, nitty-gritty optimizations that lock assumptions into your code until as late as possible.

  • Move quickly to explore your game's design space, but don't go so fast that you leave a mess behind you. You'll have to live with it, after all.

  • If you are going to ditch code, don't waste time making it pretty. Rock starts trash hotel rooms because they know they're going to check out the next day.

  • Most of all, if you want to make something fun, have fun making it!
