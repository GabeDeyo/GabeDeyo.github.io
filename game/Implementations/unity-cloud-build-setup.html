Introduction

  • Unity Cloud Build watches your source code and automatically compiles, deploys, and tests new changes.

    Git is a popular version control system that allows you to track changes to your project and revert them when needed.

    Together, they make the perfect toolset for speeding up Unity game development.

    In this section, we'll look at setting up Unity Cloud Build projects, tag builds, push changes via Git to Bitbucket, fix bugs, and manage project changes.

    Later we'll see how to optimize your workflow to get the most out of the toolset and compile games across multiple platforms: WebGL, desktop, and mobile.

Repo Setup

  • Git Setup

    Open GitBash in the project you want to setup in UCB.

                git init

  • User Setup

    Git needs to know your username and email.

                git config --global user.email "gabedeyo@gmail.com"
            git config --global user.name "Gabe Deyo"

  • .gitignore

    Exclude files getting pushed to the repo using the .gitignore file.

    Save .gitignore into the root of your project folder.

                /[Ll]ibrary/
            /[Tt]emp/
            /[Oo]bj/
            /[Bb]uild/
            /[Bb]uilds/

            *.csproj
            *.unityproj
            *.sln
            *.suo
            *.tmp
            *.user
            *.userprefs
            *.pidb
            *.boo

  • Add Changes

    Commit all changes

                git add .
            //or
            git add --all

    Commit specific changes

                git add boner.txt

    Check git status

                git status

  • Commit Changes

    Make sure all changes that you want committed are added with git status.

                git commit -m "Commit Message"

  • Bitbucket Repo

    Create a new repo in your Bitbucket account.

    Link your local project and Bitbucket repo:

                git remote add origin git@bitbucket.org:accountname/reponame.git

    Head to SourceTree and add this project path.

    Use source tree to push your commit to the repo.

Unity Cloud Build

  • Head to the UCB website - Unity Cloud Build

    Login or Signup

  • Create

    Click "Create New Project" from the Dashboard.

    We need to connect this to the source control repo.

    Grab the URL from the Bitbucket repo and paste into the Server URL field.

    Select "GIT" as the SCM Type.

    Move on to "Access".

  • SSH Key

    Since this is the first time accessing the repo, UCB will generate a unique SSH Key for you.
      • You won't have to do this if you've already set up a key.

    Copy the key contents, and got back to Bitbucket.

    Head to the Bitbucket Setting (not project settings) and go to SSH keys under "Security".

    Press Add Key and use "Unity Cloud Build" for the Label and paste in the SSH key.

  • Target Setup

    Select your initial build, preferably WebGL, as your starting build.

    Next, fill out most of the obvious details to setup the project.

    An important field to get correct is the Branch. UCB will monitor whichever branch is selected here. This enables UCB to start a build whenever changes are detected.

    It will be important down the road to make most changes in sub-branches which are then marged to the master.
      • If you always have UCB pointing to the master branch, it will only need to make builds when major changes are merged.

    After completing these, the next step will automatically build your project.

    If you're interested in more than just one target platform, head to the "Configure" tab.
      • Here you can edit your current build, add a new build setup, or build to a new target.
      • All the invidividual builds will have their own unique settings for building.

  • Run Project

    If your game is built as a WebGL build, you can simply play the game in the browser by selecting "Play".
      • There may be errors in Safari.

    Other build types will have their own options to download and play.

    In order to allow other people to play the build, simply click the "Share" button.
      • This will generate a link that anybody can use to play the build.

    Make sure to test the shareable link to make sure the game is prepared for the build.
      • If greeted with a black screen, you know there is some scene setup that still needs to be done.

Manage Project Changes

  • Branching

    A branch is a copy of the master code that you can work in safely.

    Branches can be merged back into their master branch.

    Branches are a good way to experiment with ideas, add new features, or fix bugs.
      • They are also useful when working in a team, where each member can work concurrently in the same project.
      • This allows main developers to hand pick changes they need from other people's branches, and merge them into the master branch.

  • Create Branch

    In order to create a new branch, open a terminal at the root of your project.

    git branch branch-name - The branch name should usually start with a version number and end with the description.
      • Example: git branch v0.2.0-features

    Then each time you work on a new feature or fix a bug, create a new branch and increment the version.

    This will help organize the project into a logical progression.

    Check which branch you're in with git branch.

    Head to your new branch with git checkout branch-name.

  • Merge Branch

    Before merging a branch back into the master trunk, Quit Unity. This will prevent file corruption and Unity crashing.

    Make sure the changes you want are added and committed.

    Switch back to the master branch: git checkout master.

    When ready to merge: git merge branch-name.
      • Check that there are no stray files with: git status.

    Push these changes to the repo. Remember - this will automatically trigger a build on UCB.
      • You could always manually trigger UCB to build your project.

  • Tag Projects

    A tag is a type of snapshot we can take of a project.

    Tags are generally used as a way of saving the state of a project that's finished or merged back to master.

    A typical time to create a tag is when features or fixes are merged into the master trunk of your project.

    It's standard for a new branch to be created with a new version number after a tag has been created.

    Tag a project with: git tag version-number.

    Push tags with: git push --tags

  • Workflow

    The typical workflow is: new branch -> game edits -> merge branch -> add tag -> repeat

    It's good practice to follow this workflow to have consistent and manageable version control from the beginning.

    It's also good practive to never work in the master branch. Always work in a separate branch which can safely be merged to the master branch.

    Whenever someone pulls from your repo, they will automatically pull the master branch. Help them approach the project easier by always providing a stable master branch.
